## TL;DR

- **3 tablas completamente muertas** que se pueden borrar (0 filas, 0
  referencias en código): `recommendations`, `monthly_snapshots`,
  `user_behavior_metrics`.
- **2 tablas semi-vivas** (leídas por la app pero nadie escribe): `user_memory`,
  `user_insights`. Decisión: mantener pero crear el job que las llene, o
  borrarlas. Hoy son ruido.
- **~12 columnas zombi** en `profiles`, `transactions` y `pockets` (dead
  metadata de iteraciones viejas). Ver tabla §4.
- **6 RPCs huérfanos** que nadie llama desde cliente ni Edge Functions:
  `get_financial_health`, `get_monthly_trends`, `get_pocket_real_balance`,
  `get_ai_ready_context`, `get_user_behavior_summary`, `update_save_streaks`.
- **1 función huérfana** completa: `tg_income_emit_event` está definida
  pero ningún trigger la usa.
- **🔴 Riesgo de seguridad real:** todas las RPCs `SECURITY DEFINER` están
  expuestas a `anon` por defecto vía REST — incluyendo `register_expense`,
  `register_income`, `transfer_between_pockets` y `delete_transaction_with_reversal`.
  Esto significa que cualquiera con la URL puede mutar datos de otro usuario
  pasando un `p_user_id` arbitrario, **siempre que las RPCs no validen
  `auth.uid() = p_user_id` adentro** (hay que auditar una por una).
- **Performance lineal con la base de usuarios:** 19 políticas RLS llaman
  `auth.uid()` directo en lugar de `(select auth.uid())`. A escala mata el
  query plan. Fix masivo y reversible.
- **2 FKs sin índice de soporte real:** `pockets.user_id` y
  `transactions.user_id`. Hoy son las 2 columnas más usadas en filtros.
- **3 índices duplicados/redundantes** en `user_spending_rules`.

## 8. RLS — performance y limpieza

### 8a. `auth.uid()` no envuelto en SELECT (19 políticas)

Cada política RLS llama `auth.uid()` por **fila** evaluada. A escala
(cuando un usuario tenga miles de transactions) esto multiplica el costo.
Fix: envolver en subquery `(select auth.uid())` que se evalúa una sola vez.

Tablas afectadas: `profiles`, `transactions`, `pockets`,
`recommendations`, `user_events`, `user_spending_rules`,
`monthly_snapshots`, `user_behavior_metrics`, `chat_messages`,
`user_memory`, `user_insights`.

Fix masivo en una sola migración: DROP + RECREATE con `(select auth.uid())`.

### 8b. Políticas duplicadas en `user_events`

```
"Users can manage their own events"   ALL    auth.uid() = user_id    ← vieja
user_events_select_own                SELECT auth.uid() = user_id    ← nueva
user_events_insert_own                INSERT auth.uid() = user_id    ← nueva
```

Las dos coexisten y el advisor lo flagea. **DROP** la vieja
"Users can manage their own events"; las dos nuevas la cubren.

### 8c. Políticas de `monthly_snapshots`, `user_behavior_metrics`,
`recommendations` — desaparecen al borrar las tablas.

## 9. Seguridad

### 9a. 🔴 Crítico: SECURITY DEFINER + EXECUTE público

12 RPCs son `SECURITY DEFINER` y están expuestas vía REST tanto a
`anon` como a `authenticated`:

```
register_expense        register_income       transfer_between_pockets
delete_transaction_…    get_monthly_state     get_financial_health
get_monthly_trends      get_pocket_real_…     get_user_behavior_…
get_ai_ready_context    cron_expire_…         cron_prune_…
emit_user_event         rls_auto_enable       tg_*_emit_event
```

**El riesgo concreto** es que `register_expense(p_user_id := <ALGUIEN>, …)`
puede llamarse sin login. Si la RPC NO valida `p_user_id = auth.uid()`
adentro, cualquiera con la URL puede meterle gastos a otro usuario.

**Fix:**

1. Las RPCs de mutación (`register_*`, `transfer_*`, `delete_*`):
   - Cambiar la firma para NO recibir `p_user_id` y obtenerlo de
     `auth.uid()` adentro. **O**
   - Mantener `p_user_id` pero al inicio: `IF p_user_id <> auth.uid()
     THEN RAISE EXCEPTION '42501 forbidden'`.
2. `REVOKE EXECUTE ... FROM anon` para todas las RPCs que mutan.
3. Las cron RPCs: `REVOKE EXECUTE FROM anon, authenticated` y dejarlas
   solo callable desde el cron (rol `service_role`).

### 9b. Search path mutable (12 funciones)

Warning genérico: el search_path puede ser secuestrado por un schema
malicioso. Fix uniforme: agregar `SET search_path = public, pg_temp`
en cada `CREATE FUNCTION` que mutamos.

### 9c. Auth — Leaked Password Protection desactivado

Dashboard de Supabase → Auth → Password security → activar HIBP check.
1 click, 0 código.
